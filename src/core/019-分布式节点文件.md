# 分布式节点文件——让dt节点就近存放在项目目录中

## 已知前提

用户指出旧版 dt 的不满意点：所有认知节点都集中在 `.dt/nodes/`，AI 想理解项目时必须先进入 dt 这个抽象空间；但真实工作通常按业务文件夹、实现方案和模块目录组织。认知节点应该可以跟相关实现就近存放。

历史背景：改造前旧实现的限制是：

- 项目发现：`findDtRoot()` 从当前目录向上找 `.dt/tree.yaml`，因此一个 dt 项目被定义为“某个目录下存在 `.dt/`”。
- 节点存储：节点正文集中在默认节点目录中，`listAllNodes()` 只发现这一处。
- 树结构来源：`buildForest()` 只根据节点 frontmatter 里的 `edges` 和 `root` 字段构建树。
- 多项目能力：已有 `~/.dt/projects.json` 注册表和 `projectId::nodeId` 跨项目引用，但目标项目内部也曾依赖默认节点目录解析。

当前已落地状态：

- `.dt/` 保留为项目配置与索引中心，`.dt/index.yaml` 记录 `node id -> Markdown path`。
- 节点 Markdown 已经可以放在项目任意目录；本项目已按新版结构迁移为 `项目思想/`、`src/core/`、`src/commands/`、`src/server/`、`frontend/src/` 等分层。
- `dt tree/show/status/add/update/link` 会自动刷新已知索引；`dt scan` 用于全量发现与修复。
- `dt serve` 和桌面入口会监听项目根目录下的 Markdown 与目录变化，自动刷新索引并推送前端。

用户修正了一个关键误解：这里不是要让业务源码/普通文件变成 dt 节点，也不是让 `.dt/nodes` 里的节点引用业务文件；而是 **DT 认知节点 Markdown 文件本身可以放在任意文件夹里**。

## 认知起点

这里讨论的核心问题是：

**如何让 dt 支持分布式节点文件：认知节点 Markdown 可散落在项目任意目录中，同时仍能被全局发现、索引、连边、渲染和跨项目引用？**

边界：

- `.dt/` 仍然存在，但角色从“所有节点正文的仓库”变成“项目级索引与配置中心”。
- 节点文件可以是普通 `.md`，不强制使用 `.dt.md` 后缀。
- 是否为 DT 节点由 Markdown 开头 YAML frontmatter 判断，新格式使用 `dt: node/v1` 显式标记。
- 树的逻辑结构仍由 `edges/root` 决定；文件夹位置只是节点的物理位置，不能替代认知关系。
- `.dt/index.yaml` 做 `nodeId -> path` 缓存；常规命令自动刷新索引，`dt serve` 监听文件系统变化，`dt scan` 用于显式修复/重建。

## 推导逻辑

### 存储模型

```text
project/
  .dt/
    tree.yaml
    index.yaml        # node id -> Markdown path
    nodes/            # 兼容旧默认位置

  src/auth/
    login-design.md   # 可作为 dt 节点

  frontend/login/
    interaction.md    # 可作为 dt 节点
```

节点身份在 frontmatter 中，而不是文件名中：

```yaml
version: 1
nodes:
  "019":
    id: "019"
    path: docs/design/distributed-node.md
    mtimeMs: 123456
    hash: ...
```

### 节点识别

不使用文件名后缀作为识别条件。扫描 `.md` 文件时读取 frontmatter：

```yaml
---
dt: node/v1
id: "021"
title: 登录鉴权设计
summary: ...
type: explore
status: pending
root: false
edges: []
created: ...
---
```

旧节点没有 `dt` 字段但具备完整 dt frontmatter 结构时仍兼容，并在读取时惰性补上 `dt: node/v1`。

### 新建节点

新增节点可以显式指定路径：

```bash
dt add explore "登录鉴权设计" --from 002 --path src/auth/login-design.md
dt add explore "登录鉴权设计" --directory src/auth
```

`--path` 是精确文件路径；`--directory` 表示写入指定目录，默认文件名为 `<id>.md`，也可以用 `--filename <name>` 指定文件名。不传路径或目录时继续写入 `.dt/nodes/<id>.md`，保证旧工作流可用。

### 读取与更新

常规读取会自动刷新索引：

- 读 `.dt/index.yaml`
- 校验已索引路径是否还存在、是否仍是 DT 节点
- 兼容性发现 `.dt/nodes/*.md`
- 需要完整发现/修复时执行全量扫描
- 显示节点路径

如果用户手写了新的 DT Markdown，常规命令会自动刷新已知索引；需要全量发现或修复时运行：

```bash
dt scan
```

做全量扫描，将符合 schema 的 Markdown 纳入索引。

### 跨项目引用

跨项目引用仍然用稳定节点 ID，而不是路径：

```yaml
edges:
  - target: other-project::019
    type: from
    summary: 依赖另一个项目的设计结论
```

解析流程：

1. 查 `~/.dt/projects.json` 找到 `other-project` 的项目路径。
2. 读取该项目 `.dt/index.yaml`。
3. 通过 `019` 找到真实 Markdown 路径。
4. 读取节点并维护双向边。

因此全局 `~/.dt` 只是项目目录服务；局部项目 `.dt` 才是该项目的权威索引。

## 讨论共识

已落地的方向：

- 节点 Markdown 可以放在项目任意目录。
- `.dt/index.yaml` 保存 `id -> path` 索引。
- 新节点 frontmatter 使用 `dt: node/v1` 明确标记。
- `dt add --path <path>` 支持指定节点文件路径。
- `dt add --directory <dir> [--filename <name>]` 支持按目录新建节点文件。
- `dt scan` 支持全量扫描项目 Markdown 并发现 DT 节点。
- `dt tree` 会显示未挂接节点，避免“先创建后连边”的节点在 CLI 里隐身。
- `dt tree/show` 展示每个节点的实际路径。
- `dt serve` 监听项目 Markdown 和目录变化并刷新索引。
- 前端 Project Explorer 展示项目文件树，并支持在指定目录创建节点/文件夹。
- 跨项目引用保持 `projectId::nodeId`，再由目标项目索引解析真实路径。

后续可以继续讨论：

- 前端是否增加路径选择/移动节点文件的交互。
- 是否需要 `dt move <id> <path>` 专门移动节点并更新索引。
- `dt scan` 的冲突处理策略：同一个 id 出现在多个 Markdown 时，应提供更明确的修复命令。
