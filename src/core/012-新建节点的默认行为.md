# 新建节点的默认行为——先创建，再建立关系

## 认知起点

用户修正了旧理解：新建节点不应该总是被迫绑定到某个父节点。真实工作里，人经常会先捕捉一个认知点、方案点或待整理问题，然后再判断它和其他节点的关系。

因此这里讨论的不是“如何禁止孤立节点”，而是：

- 如何让节点可以先在当前位置被创建出来；
- 如何让人随后用拖拽、详情面板或 `dt link` 补关系；
- 如何避免 AI 被少数固定模板限制住。

## 推导逻辑

### 创建与连边是两个动作

旧逻辑把“创建节点”和“声明关系”绑定在一起，优点是树更整齐，但缺点是会打断探索过程。新的逻辑把它们拆开：

- `dt add <type> "标题"`：创建一个未挂接节点；
- `dt add <type> "标题" --from <id>`：创建时同时声明父节点；
- `dt add <type> "标题" --root`：创建根节点；
- `dt link <src> <tgt> "摘要"`：在已有节点之间补关系；
- 前端画布可以先新建节点，再从节点 handle 拉线建立关系。

这样“先有想法、再组织结构”的工作方式可以成立。

### 新建位置由文件系统上下文决定

分布式节点引入后，新建节点还需要回答“文件写在哪里”：

- CLI 精确控制：`--path <path>` 写到指定 Markdown；
- 前端默认控制：左侧 Project Explorer 当前选中的目录就是新节点落点；
- 如果选中的是某个 DT Markdown，则默认落到同目录；
- 如果选中的是 `.dt/nodes` 里的旧节点，为避免继续把分布式节点写回 `.dt`，前端默认落到项目根目录。

`.dt/nodes` 仍然是兼容默认位置，但新的分布式节点更推荐放在和实现/讨论上下文相近的目录里。

### 模板是参考，不是限制

四类常见写作风格（explore / task / document / decision）只是全局 `$dt` skill 里的参考模板。节点类型是自由字符串，正文内容也不被强制预填。

这解决两个问题：

- AI 不会被限制成只能使用四种类型；
- 团队可以逐步沉淀新的全局 skill/template，而不是在每个项目里重复维护。

## 讨论共识

- 新节点允许先创建为未挂接节点，之后再补关系。
- 前端支持在当前目录或新建目录中创建节点。
- 前端画布提供“节点”和“根节点”两个快速创建入口。
- 关系建立通过画布拖拽、详情面板添加连接或 `dt link` 完成。
- 节点类型与正文结构保持开放；全局 `$dt` skill 里的模板只是参考。
