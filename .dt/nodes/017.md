# 受众定义与痛点分析

## 已知前提

- 现有 README 让人不知道工具的实际价值在哪里
- 现有介绍太抽象、太技术，看了不知道自己是否该用、如何用
- 最好的介绍方式一定要从**痛点出发**，而不是从工具功能出发
- 不同场景（README / 仓库描述 / 技术博客）对应不同的介绍深度

## 认知起点

在设计项目介绍之前，必须先回答：**我在对谁说话？他的痛是什么？**

介绍的目标不是"解释清楚 dt 是什么"，而是"让对方感受到：这是我一直需要的东西"。

边界：本节点只讨论受众分类和痛点定义，不讨论具体写法。

## 推导逻辑

### 受众分类

**群体 A：开发者**
- 背景：熟悉 CLI、Git、AI 辅助编程
- 典型场景：用 Claude Code / Cursor 开发，但经常遇到 AI "跑偏"、"做多了"、"下一个会话什么都忘了"
- 核心痛点：
  - 跨会话上下文断裂——每次新对话都要重新交代背景
  - AI 推进速度太快，来不及确认每个假设是否对齐
  - 最终产物是 AI 的，不是自己的——改动时不知道为什么这样做

**群体 B：AI 重度用户（非技术背景）**
- 背景：每天用 AI 辅助思考/写作/规划，但不写代码
- 典型场景：用 ChatGPT / Claude 做策略讨论、产品规划
- 核心痛点：
  - AI 回答太快，没有给自己留下思考的空间
  - 描述不清楚时 AI 会自行填充假设，但用户不知道假设在哪里
  - 讨论结论有了，但推导过程没有留存，下次找不到当时为什么这样决定

**群体 C：GitHub 开源社区路人**
- 背景：快速扫描，5 秒内判断要不要继续读
- 核心需求：一句话能抓住他们——要么痛点精准，要么概念新颖

### 现有 README 的问题

- 第一句"决策过程外化工具"——**太抽象**，不知道这是什么体验
- "人与 AI 之间只有语言这一个通道"——**哲学层太早出现**，在没有感受到痛之前，读到哲学会让人感到"在讲大道理"
- 快速开始直接给命令——**对群体 B 完全不友好**

### 真实痛点来源（用户自述）

1. **人工构建层级上下文很困难**：你可以手动建 Markdown 文件 + 文件夹来模拟这个结构，但你需要自己做认知划分、自己维护层级关系、自己管理"哪个结论来自哪个推导"——这件事本身就是认知负担。

2. **AI 生成速度太快，隐藏认知没有同步**：我有很多没说出口的假设、边界、判断标准，AI 不知道。但它不会停下来问，它会自信地往下走。

3. **使用 AI 的过程中失去了思考**：希望保持对关键逻辑的思考，而不是把所有判断都外包给 AI，导致最后自己对项目没有掌控感——别人问起来，只能再去让 AI 解释。

4. **思维图构建完成后可以被多个 Agent 调用**：当决策树足够完整时，可以作为结构化上下文驱动多个并行的 Agent 任务——这是从个人工具到工作流基础设施的升级。

### 与其他工具的对比

| 工具 | 能做到的 | 做不到的 |
|---|---|---|
| Markdown + 文件夹 | 本地、可读、可 Git | 需要手动维护结构，无 AI 集成 |
| Obsidian | 双链笔记、可视化 | 不是为决策推导设计，无 Agent 集成 |
| Notion / 思维导图 | 结构可视化 | 不本地、无 CLI、无 Agent 协议 |
| 直接用 Claude / ChatGPT | 生成快 | 过程不留存、跨会话断裂、用户失去思考主导权 |
| **dt** | 本地 Markdown + CLI + Agent 协议 | 强迫双方慢下来对齐，过程可追溯 |

**dt 的本质差异**：不是更好的笔记工具，而是**把推导过程变成结构化协作协议**——让 AI 和人共享同一份"为什么"，而不只是"是什么"。

## 讨论共识

- ✅ 受众分三类：开发者、AI 重度用户（非技术）、GitHub 路人
- ✅ 介绍必须从痛点出发，不从功能出发
- ✅ 核心痛点：上下文断裂、对齐失效、失去掌控感、思考被外包
- ✅ 与其他工具的本质区别：dt 记录的是"为什么"，不是"是什么"
- ✅ 下一步：为每个场景（README / 仓库描述 / 技术博客）分别建子节点，对应不同受众和介绍深度
