# 为什么要有dt这个项目

## 已知前提
- 现有模型能力已经足够强，并行执行能力很强
- 所有主流 Agent 都支持 Bash 命令操作
- 人类的认知能力有限，无法追溯几万行代码的生成过程
- Obsidian Canvas 无法满足决策过程可视化的需求
- 现有 AI 工作流是"给任务 → 执行"的单向模式

## 推导逻辑

### 核心矛盾
瓶颈不在 AI 能力，而在**人机接口**。具体表现为：

| 现象 | 本质 |
|---|---|
| Agent 输出是"黑盒结果"，无法追溯 | 缺少**过程透明度** |
| 几万行代码，人无法审查 | 人机**产出与审查能力不匹配** |
| 人类给 Agent 的描述不准确 | 人机之间缺少**共享心智模型** |
| 人无法有效参与 Agent 决策过程 | 缺少**人在回路**的结构化机制 |

### 深层问题

#### 1. 探索性任务天然不清晰
现有 AI 工作流假设人类能给出明确任务，但现实中很多工作本身就是探索过程——人类自己也不清楚最终要什么。需要在推导过程中逐步明晰目标。

#### 2. 人机认知未对齐
人和 AI 对同一事实的理解可能存在偏差。如果没有在前提层面达成共识，后续的方案推导和选择就会产生系统性偏差。**必须先对齐认知，再做决策。**

#### 3. AI 执行速度放大了偏差
AI 并行能力太强，需求稍有偏差就一口气做完大量错误的工作。速度越快，纠错成本越高。因此需要在关键节点设置检查点，而不是等最终结果。

### 解决思路
用**决策树**外化 Agent 的思考过程：
1. 每一步推导都有**已知前提、推导逻辑、备选方案、方案评估**
2. 人类可以在任意节点介入、查看、修改、确认
3. 决策过程可视化，人类可以追溯和理解
4. 通过 CLI 实现，任何 Agent 都能使用，零集成成本

### 为什么是决策树？
- 人类在真实世界中就是通过分支探索和权衡取舍来做决策的
- 线性对话流缺乏结构化的分支探索和回溯能力
- 决策树天然支持：目标拆分 → 方案并行 → 评估选择 → 递归深入

### 为什么是 CLI + 本地文件？
- **CLI 颗粒化**：所有 Agent 都能用 Bash，CLI 命令是最通用的接口
- **讨论过程本地化**：所有决策过程沉淀为本地 Markdown 文件，任何新的 AI Agent 加入时，只要读这些文件就能了解之前讨论了什么、达成了什么共识，实现**上下文的跨会话、跨 Agent 传递**
