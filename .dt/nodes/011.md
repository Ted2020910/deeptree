# CLAUDE.md协议优化——如何引导Agent正确使用dt

## 已知前提

- CLAUDE.md 是 Agent 收到的系统指令，直接影响 Agent 的行为模式
- 现有 CLAUDE.md 内容：工具说明（命令表）+ 工作方式（看/想/写）+ 节点设计原则
- 核心问题：**只告诉了 Agent 有什么工具，没有告诉 Agent 怎么用、什么时候用、用到什么程度**
- Agent 的行为受提示词驱动，提示词越具体，行为越可预期

## 当前内容的缺口

### 1. 缺少"何时启动 dt"的判断标准

现在只说"如果存在 .dt/ 项目，先执行 dt tree"。但没有说：
- 什么类型的任务需要用 dt 记录？
- 什么类型的任务不需要（直接回答即可）？
- Agent 自己判断还是等用户指示？

### 2. 缺少会话工作流

"先执行 dt tree"之后做什么？没有描述：
- 如何从树的状态判断当前任务在哪个节点
- 如何决定是继续已有节点还是创建新节点
- 何时应该暂停询问用户，而不是自行推进

### 3. 缺少"记录什么"的指导

Agent 可以自由编辑正文，但没有指导：
- 正文应该记录推导过程，还是只记结论？
- 讨论未达成共识时怎么记？
- 用户的关键反馈怎么处理？

### 4. 缺少节点类型使用场景

只列了预设类型名称，没有说：
- goal/subproblem/solution/evaluation/reflection 各自代表什么情景
- 什么时候用自定义类型
- 节点粒度怎么把握（太细 vs 太粗）

### 5. 缺少认知对齐动作

dt 的核心目的是认知对齐，但 CLAUDE.md 没有指导：
- 何时应该把树结构展示给用户（"我们现在在 003 节点讨论..."）
- 如何通过节点标题+摘要让用户快速确认方向
- 用户的否定/修正如何映射到树的更新

## 讨论方向

### 方向一：工作流协议

给 Agent 一个完整的工作循环：

```
会话开始 → dt tree（了解现状）
↓
用户提出问题/任务
↓
判断：探索型（需要 dt）or 确定型（直接回答）？
↓（探索型）
定位到树中的节点，或创建新节点
↓
拆解 → 分析 → 关键决策点暂停
↓
记录讨论共识到节点
↓
更新节点状态
```

### 方向二：何时用 dt 的判断规则

明确触发条件：
- **用 dt**：任务有不确定性、需要拆解、涉及多方案权衡、需要跨会话持续
- **不用 dt**：事实性问题、单步操作、用户明确说"直接做"

### 方向三：正文写作规范

轻量引导而非强制模板：
- 推导过程 > 结论列表（结论没有推导过程，下次看不懂）
- 记录"为什么这样决定"比记录"决定了什么"更有价值
- 用户的关键修正要显式记录（"用户指出..."）

### 方向四：认知对齐动作

Agent 应该主动：
- 说出当前在哪个节点（"我们现在讨论节点 003..."）
- 在关键分叉点呈现选项让用户选
- 完成一个节点时更新状态并报告

## 待讨论

1. CLAUDE.md 应该多长？太长 Agent 不会读，太短没用
2. 工作流协议是否应该直接写进 CLAUDE.md，还是另外维护？
3. 不同类型项目（代码 vs 策略规划）需要不同的协议吗？
