# 新建节点的默认行为——关系绑定与正文结构

## 认知起点

- 节点是决策过程的载体，孤立节点没有上下文，无法被理解
- 现有 `dt add` 只强制要求 type 和 title，parent 是可选的
- 节点正文目前只生成 `# {title}`，Agent 自由发挥
- 核心矛盾：自由度太高 → Agent 生成无结构节点；结构太死 → 失去灵活性

## 推导逻辑

### 问题一：非孤立节点必须带关系

**什么是"孤立节点"**：既没有 parent，也没有 edges 的节点。

孤立节点在 dt 中几乎没有意义：
- 没有关系，就没有上下文（这个节点从哪来？为什么存在？）
- Agent 在树中看到孤立节点，无法判断它的位置和重要性
- 孤立节点破坏树的可追溯性

期望行为：
- 有 `--parent`：正常创建（已实现）
- 没有 `--parent`：仅允许创建项目首个 goal 根节点，其余情况强制要求关系
- `dt add` 增加 `--link <id> "摘要"` 选项，支持在创建时同时绑定跨分支关联（否则需两步：add + link）

### 问题二：正文默认结构

核心目的：体现"从什么认知基础出发 → 经过什么推导 → 得到什么共识"。

三层骨架：

| 层 | 名称 | 内容 |
|---|---|---|
| 起点 | 认知起点 | 已知前提、约束、背景共识 |
| 过程 | 推导逻辑 | 分析、拆解、方案比较、权衡 |
| 终点 | 讨论共识 | 最终结论或决策（讨论中可为空） |

**为什么不是强制模板**：强制填充空 section 是形式主义。应该是默认骨架 + 鼓励填充，非必填。

期望的默认正文：
```markdown
# {title}

## 认知起点

## 推导逻辑

## 讨论共识
```

与 CLAUDE.md 的关系：节点骨架本身就是写作指引——Agent 看到这三个 section，自然知道从认知起点开始写，分析写入推导逻辑，达成共识后更新第三节。不需要 CLAUDE.md 里再单独解释"正文应该写什么"。

## 讨论共识

- ✅ `dt add` 支持 `--link <id> "摘要"` 选项，创建时同时绑定跨分支关联
- ✅ 没有 parent 时不做任何限制，CLAUDE.md 中说明即可（孤立节点由 Agent 自律）
- ✅ 节点正文默认四层结构：**已知前提 → 认知起点 → 推导逻辑 → 讨论共识**
- ✅ CLAUDE.md 中需要额外解释节点正文的写作规范
