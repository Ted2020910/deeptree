# CLI命令设计——如何让Agent准确使用

## 已知前提
- Agent 通过 Bash 调用 CLI，这是唯一接口
- Agent 的上下文窗口有限，CLI 描述和输出要尽可能简洁
- Agent 不需要理解 CLI 的实现细节，只需理解：效果 + 工作模式
- Agent 的工作模式本质上是：拆解 → 分析 → 交互 → 反馈

## 推导逻辑

### 核心矛盾
CLI 命令太多 → Agent 难以准确选择，上下文被占用
CLI 命令太少 → 表达能力不够，无法覆盖所有操作

### 五个动作模型

Agent 使用 dt 时，本质上只在做五件事：**看、想、问、写、听**

---

#### 1. 看（了解现状）

核心问题：**Agent 看到的信息量应该是多少？**

分层设计：
- **CLI 直接输出**：树的结构概览（标题 + 摘要 + 类型 + 状态），直接进入上下文
- **Agent 自主检索**：如果需要深入某个节点，Agent 可以用 Read 工具读取节点的 Markdown 文件
- **文件链接**：节点中关联的详细文档，Agent 按需 Read

> 原则：CLI 输出是"目录"，具体内容让 Agent 自己 Read。节约上下文窗口。

对应命令：
```bash
dt tree              # 看全局结构（标题+摘要+状态）
dt status            # 看项目概览
dt show <id>         # 看某个节点详情（输出 frontmatter + 正文摘要）
```

Agent 要看更深的内容？直接 `Read .dt/nodes/003.md`，不需要额外 CLI 命令。

---

#### 2. 想（拆解分析）

对应操作：添加节点、添加边

```bash
dt add <type> "标题" --parent <id>     # 添加节点
dt link <source> <target> "逻辑摘要"    # 添加边（跨分支关联）
```

---

#### 3. 问（交互确认）

**不需要专门的 CLI 命令。砍掉 `dt ask`。**

理由：
- Agent 在终端中本来就是和用户对话的，直接用自然语言提问
- 前端的角色是**可视化文件结构**，不是交互界面
- 人的交互场景始终在终端，前端只是"看板"

---

#### 4. 写（记录结论）

**采用折中方案**：

| 内容 | 方式 | 理由 |
|---|---|---|
| **frontmatter**（结构化字段） | CLI 命令更新 | 保证格式正确，系统可追踪 |
| **Markdown 正文**（自由内容） | Agent 直接 Edit 文件 | 自由度高，零学习成本 |

```bash
dt update <id> --status completed      # 更新结构化字段
dt update <id> --summary "新摘要"       # 更新摘要
dt update <id> --type evaluation       # 更新类型
# 正文内容 → Agent 直接 Edit .dt/nodes/003.md
```

---

#### 5. 听（感知用户变更）——参考 Git 的思路

核心问题：**如何感知用户对文件进行了修改？**

##### 直接使用 Git，不造轮子——且对 Agent 透明

不需要自己实现快照机制——**直接用 git 本身**。

但关键原则：**Agent 不需要知道 git 的存在。所有 git 操作由 dt CLI 内部自动完成。**

dt CLI 的每个写操作（add / update / link）执行完后，内部自动：
```
1. 修改 .dt/ 文件
2. git add .dt/
3. git commit -m "dt: <操作描述>"
```

Agent 只看到 `dt add subproblem "xxx" --parent 002` 成功了，不知道背后发生了 git commit。

同样，当 Agent 直接 Edit 了 .dt/nodes/xxx.md 的正文后，下一次调用任何 dt 命令时，CLI 可以自动检测到 unstaged changes 并 commit。

##### 区分人类编辑和 AI 编辑

核心思路：**dt CLI 操作后自动 commit，工作区始终"干净"。**

时序逻辑：
```
Agent: dt add subproblem "xxx"
  └── CLI 内部自动 git add + commit → 工作区干净

Agent: Edit .dt/nodes/003.md（直接编辑正文）
  └── 下次调用 dt 命令时，CLI 发现 unstaged → 自动 commit → 工作区干净

AI 空闲（等待用户输入）
  └── 工作区是干净的
  └── 人类编辑（Obsidian/编辑器/前端）→ 产生 unstaged changes

Agent 下次调用 dt tree
  └── CLI 内部 git diff .dt/ → 发现 unstaged changes = 全部是人类修改
  └── 输出变更提示
```

**Agent 全程不需要执行任何 git 命令。git 是 dt CLI 的内部实现细节。**

##### 自动检查——嵌入 dt tree

不需要 Agent 单独调用 diff，**把变更检查嵌入到 `dt tree` 中**：

```bash
dt tree
# 输出：
# 🌳 dt项目设计
# ⚡ 检测到 2 处用户修改:
#   - 003.md: 正文被修改（推导逻辑部分）
#   - 新增 010.md
# ────────────────────────
# ⏳ 002 [goal] 为什么要有dt这个项目
# ├── ⏳ 003 [subproblem] 人类真实的工作决策模式是什么 ✏️
# ...
```

Agent 在正常工作中自然会调用 `dt tree` 查看全局结构，这时**顺带**告诉它有什么变更。无需额外调用，无需反复轮询。

同理，`dt status` 也可以附带变更检查。

##### dt listen --daemon → 不再需要

既然：
- Agent 工作时：`dt tree` / `dt status` 自动检查变更
- Agent 空闲时：人类编辑后，下次对话 Agent 自然会 `dt tree`

就不再需要后台 daemon 监听了。整个"听"的机制大幅简化。

## 讨论共识
- ✅ 五个动作模型（看/想/问/写/听）是 CLI 设计的骨架
- ✅ "看"分层：CLI 输出概览，深入内容让 Agent 自己 Read 文件
- ✅ "问"砍掉 `dt ask`，前端是看板不是交互界面，交互在终端
- ✅ "写"折中：frontmatter 走 CLI，正文 Agent 直接编辑
- ✅ "听"直接用 git diff，不造快照轮子
- ✅ **git 操作对 Agent 完全透明**：dt CLI 每次写操作后自动 git add + commit，Agent 不接触 git
- ✅ Agent 直接 Edit 正文后，下次 dt 命令自动补 commit
- ✅ 工作区 unstaged changes = 人类修改（因为 AI 的修改已被自动 commit）
- ✅ 变更检查嵌入 `dt tree` / `dt status`，Agent 无需额外调用
- ✅ 砍掉 `dt listen --daemon`，不再需要后台监听

## CLI 命令汇总（精简版）

| 动作 | 命令 | 说明 |
|---|---|---|
| **看** | `dt tree` | 全局结构 + 自动检查用户变更 |
| **看** | `dt status` | 项目概览 + 自动检查用户变更 |
| **看** | `dt show <id>` | 节点详情（frontmatter + 正文摘要） |
| **想** | `dt add <type> "标题" --parent <id>` | 添加节点（内部自动 git commit） |
| **想** | `dt link <src> <tgt> "摘要"` | 添加跨分支关联边（内部自动 git commit） |
| **写** | `dt update <id> --status/--summary/...` | 更新 frontmatter 字段（内部自动 git commit） |
| **写** | Agent 直接 Edit .dt/nodes/xxx.md | 更新正文（下次 dt 命令自动补 commit） |

共 6 条 CLI 命令 + 直接文件编辑。Agent 只需记住 6 条命令，完全不接触 git。
