# CLI命令设计——如何让Agent准确使用

## 已知前提
- Agent 通过 Bash 调用 CLI，这是唯一接口
- Agent 的上下文窗口有限，CLI 描述和输出要尽可能简洁
- Agent 不需要理解 CLI 的实现细节，只需理解：效果 + 工作模式
- 数据模型已确定：edges（from/to）替代 parent/children，支持多父和跨项目引用

## 认知起点
Agent 使用 dt 时本质上只做三件事：**看（了解现状）、想（拆解分析）、写（记录结论）**

## 推导逻辑

### 动作模型 → 命令映射

**看：**
- `dt tree`——全局结构 + 自动检查用户变更，支持多根并排
- `dt status`——项目概览（节点统计、根节点列表）
- `dt show <id>`——节点详情（frontmatter + 正文）

深入内容让 Agent 直接 `Read .dt/nodes/xxx.md`，不走 CLI，节约上下文。

**想：**
- `dt add <type> "标题" --from <id>`——添加节点，`--from` 可多次（多父），`--root` 标记根节点
- `dt link <src> <tgt> "摘要" --direction from|to`——添加边，系统自动补反向边

**写：**
- `dt update <id> --status/--title/--summary/--type/--root`——更新 frontmatter 字段
- 正文内容：Agent 直接 Edit `.dt/nodes/xxx.md`，下次 dt 命令自动补 git commit

### git 对 Agent 完全透明
- dt CLI 每次写操作后自动 git add + commit
- Agent 直接编辑正文后，下次 dt 命令检测 unstaged changes 并自动 commit
- 工作区 unstaged = 人类编辑，Agent 不接触 git

### 跨项目引用
- target 格式：`projectId::nodeId`
- `dt link 009 app-backend::015 "摘要" --depth 2`
- 项目未注册时报错，提示用户先 `dt register <path>`

## 讨论共识
- ✅ `--parent` 改为 `--from`，支持多次传入（多父节点）
- ✅ `--link/--link-summary` 废弃，统一用 `dt link`
- ✅ `dt link` 自动补反向边，方向通过 `--direction from|to` 指定
- ✅ 去掉 `related` 关系类型，只有 `from`/`to`
- ✅ 新增 `dt update --root true|false`
- ✅ 跨项目引用硬错误（未注册报错），不做软降级
- ✅ 删除 `dt sync`、`dt mirrors`——不需要镜像文件，动态解析
