# 核心数据模型——节点与边需要记录什么

## 已知前提
- dt 既给 AI 用（CLI），也给人用（前端可视化 + 直接编辑文件）
- 所有数据本地化为 Markdown + YAML frontmatter
- 需要支持多父节点、跨项目引用

## 认知起点
原有数据模型（v2）有三套冗余关系字段（parent/children/edges），且 parent 单值无法表达多父。本节点讨论如何重构为统一、简洁的数据模型。

## 推导逻辑

### 核心决策：去掉 parent/children，统一用 edges

旧模型（v2）的问题：
- `parent: string | null`——单值，无法多父
- `children: string[]`——冗余，是 parent 的反向，靠系统维护同步
- `edges: Edge[]`——有 `direction: from|to|related`，与 parent/children 语义重叠

新模型（v3）：
```yaml
id: '009'
root: false
title: '...'
summary: '...'
type: subproblem
status: pending
created: '...'
edges:
  - target: '005'
    type: from
    summary: "005 是 009 的父节点"
  - target: '007'         # 多父
    type: from
    summary: "..."
  - target: 'app-backend::015'   # 跨项目引用
    type: from
    summary: "..."
    depth: 2
```

### 边的语义
| type | 含义 |
|------|------|
| `from` | 本节点来自目标节点（目标是父节点/前提） |
| `to` | 本节点指向目标节点（目标是子节点/推导结果） |

去掉 `related`——语义不明确，改用明确的 `from`/`to` 表达。

### 双向存储，系统保证一致性
- 用户/AI 只写一侧
- 系统（addEdgeBidirectional）自动补反向边
- 写入 `from` → 自动在目标节点补 `to`，反之亦然

### 跨项目引用格式
- 格式：`projectId::nodeId`（双冒号分隔）
- 解析：查 `~/.dt/projects.json` 注册表，直接读目标项目文件
- 硬引用：项目未注册或路径不可达时报错
- 无镜像文件——动态解析，不缓存快照

### root 字段
- `root: true` 的节点作为 `dt tree` 的起始点
- 支持多根（多棵树并排渲染）
- 替代 `tree.yaml` 里的 `root_node` 字段

### tree 渲染逻辑
1. 找所有 `root: true` 节点 → 作为树根
2. 从 `to` 边递归找子节点
3. 多父节点标记 `[+]`
4. visited set 防循环引用，标记 `[循环引用]`

## 讨论共识
- ✅ 去掉 `parent`、`children` 字段，统一用 `edges`
- ✅ 边类型只有 `from`/`to`，去掉 `related`
- ✅ 双向存储：系统自动补反向边，用户只写一侧
- ✅ `root: boolean` 替代 `tree.yaml.root_node`，支持多根
- ✅ 跨项目引用 `projectId::nodeId`，硬引用，动态解析
- ✅ 旧格式惰性迁移：readNode 时自动转换，旧 related 边丢弃
