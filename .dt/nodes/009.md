# 核心数据模型——节点与边需要记录什么

## 已知前提
- dt 既给 AI 用（CLI），也给人用（前端可视化 + 直接编辑文件）
- Agent 不需要关注 CLI 本身的细节，只需理解效果和工作模式：拆解→分析→交互→反馈
- 人的认知模式分两种：总揽全局（可视化树形图）和局部细节（节点深入）
- 为节约上下文窗口，CLI 输出应尽可能简洁
- 所有数据本地化为 Markdown + YAML

## 推导逻辑

决策树的核心数据只有两类：**节点**和**边**。

### 一、节点——每个决策步骤的信息载体

一个节点需要回答的核心问题：**"这一步在讨论什么、前提是什么、结论是什么"**

节点分为两层：**结构化元数据**（YAML frontmatter）和**自由内容**（Markdown 正文）。

#### 结构化元数据（frontmatter）

这些字段是系统需要的，用于树的渲染、状态管理、CLI 操作：

| 字段 | 说明 |
|---|---|
| **id** | 全局唯一标识 |
| **标题** | 一句话概括 |
| **摘要** | 简短说明这个节点在探讨什么问题（比标题更具体，帮助人快速理解） |
| **类型** | 预设类型（goal / subproblem / solution / evaluation / reflection）或自定义类型 |
| **状态** | pending / in_progress / decided / completed / rejected |

#### 自由内容（Markdown 正文）

以下内容不做强制结构约束，由 AI 自己编辑、人自己阅读：

- 已知前提
- 推导逻辑
- 方案/结论
- 人类标注（评价、备注）
- 关联文件链接
- 任何 AI 或人觉得需要记录的内容

> **设计原则**：frontmatter 管结构，Markdown 管内容。系统只读 frontmatter，正文是人和 AI 的自由空间。

#### 渐进式披露映射
```
总揽全局 → 只看：标题 + 摘要 + 类型 + 状态
局部细节 → 展开 Markdown 正文
深层探究 → 正文中的文件链接
```

### 二、边——节点之间的关系

边需要回答的核心问题：**"为什么从 A 走到 B"**

#### 边的字段

| 字段 | 说明 | 示例 |
|---|---|---|
| **source → target** | 方向 | 002 → 003 |
| **摘要** | 一句话说明这条边的逻辑 | "从第一性原理出发" |
| **详细逻辑**（可选） | 深层拆分，链接到独立文件 | `[[edges/e001.md]]` |

边也遵循渐进式披露：
```
总揽全局 → 只看摘要
深入理解 → 通过 Markdown 链接访问详细逻辑文件
```

#### 边的存储方式：放在节点的 frontmatter 中

边信息存储在节点文件中，这样人或 AI 读一个节点时，就能立刻知道它与其他节点的所有关系：

```yaml
---
id: "003"
title: "人类真实的工作决策模式是什么"
summary: "从第一性原理出发，先理解人类如何做决策，再设计工具"
type: subproblem
status: pending
edges:
  - target: "002"
    direction: from        # 从 002 来
    summary: "从第一性原理出发，先理解决策模式才能设计工具"
  - target: "004"
    direction: to          # 到 004 去
    summary: "理解人类模式后，才能定义dt的本质"
  - target: "007"
    direction: related     # 跨分支关联
    summary: "人类决策模式是人类工作方式讨论的基础"
---
```

#### 数据结构本质：不是树，是图

> 决策过程不一定是严格的树形结构，不同分支的节点之间可能存在跨分支关联。
> 因此 dt 的底层数据结构是**有向图（DAG）**，树只是最常见的展示形态。

关系类型：
- **父子关系**（from/to）：树形主干，体现拆分和推进
- **跨分支关联**（related）：非父子节点之间的逻辑关联

#### 关于优先级/顺序

不放在边上，体现在**层级结构本身**：
- **线性关系**：A → B 意味着 A 完成后才进入 B，顺序本身就是优先级
- **并行关系**：A 的多个子节点如果没有依赖，它们天然是并行的

#### 关于关系类型标签

不需要单独的类型标签（拆分/推进/依赖）。边的"摘要"已经自然描述了关系的性质，额外标签是冗余的。

## 讨论共识
- ✅ 节点 = 结构化 frontmatter（id/标题/摘要/类型/状态/edges）+ 自由 Markdown 正文
- ✅ 新增"摘要"字段，帮助人通过标题+摘要快速理解节点内容
- ✅ 类型支持预设 + 自定义
- ✅ 正文内容不做强制结构，AI 和人自由编辑
- ✅ 边存储在节点 frontmatter 中，包含：方向 + 摘要 + 可选详细逻辑链接
- ✅ 边也支持渐进式披露：摘要 → 链接到详细文件
- ✅ 支持跨分支关联（related），底层是有向图而非严格的树
- ✅ 优先级体现在层级结构中，不放在边上
- ✅ 关系类型标签不需要，摘要已隐含
