# GitHub README 改写——让开发者和路人5秒内感受到痛点

## 已知前提

- 来自 017：受众分三类（开发者 / AI 重度用户 / GitHub 路人），介绍必须从痛点出发
- 现有 README 太抽象、太技术，看不出实际价值
- 核心痛点：上下文断裂、对齐失效、失去掌控感、思考被外包
- dt 与其他工具的本质差异：记录的是"为什么"，不只是"是什么"

## 认知起点

**三个已对齐的设计判断：**
1. 受众分层：第一屏通用痛点（不提 CLI），下方分开照顾两类人
2. 目标行为：感受到痛 → 看演示（演示展示对话过程，而非工具界面）
3. 一句话定位的重心：**强调过程**——过程被记录 → AI 表现更好 + 人对项目有掌控感

**边界：** 本节点讨论 README 的结构设计和文字草稿，不讨论演示素材制作。

## 推导逻辑

### 一句话定位的推导

候选方向三个，最终选择**强调过程**：

> 把你和 AI 的推导过程记录下来——让下一个 AI 站在你们已对齐的地方继续工作。

但这句话对 GitHub 路人还是太长。需要一个更短的版本放在 title 下面：

> **让 AI 知道你怎么想的，而不只是你要什么。**

这句话同时触达两类受众：
- 开发者读到：对，AI 经常只知道我要什么，不知道我为什么这样要——这导致它跑偏
- AI 重度用户读到：对，我每次都要重新解释背景，AI 根本不知道我的判断依据

### README 结构设计

```
第一屏（fold 以上）
├── 标题：dt
├── 一句话：让 AI 知道你怎么想的，而不只是你要什么
└── 三行痛点（无序列表，无 CLI 术语，两类人都能感受到）
    ├── 你描述需求，AI 开始生成——但它没有看到你脑子里的假设和边界
    ├── 对话结束，结论有了，但推导过程消失了——下次你自己也想不起来为什么
    └── 用 AI 做完一个项目，有人问你为什么这样做——你只能再去问 AI

第二屏：演示区（先用文字占位，后补截图序列）
└── 一段真实对话截图：问题 → AI 拆解节点 → 树形图 → 人介入修正

第三屏：这解决了什么（对两类人展开）
├── 开发者：跨会话上下文不再断裂，AI 每次从已对齐处继续
├── AI 用户：对话过程有结构，你随时可以介入、修改、确认
└── 所有人：最终产物里有你的判断，不只是 AI 的生成

第四屏：快速开始（对能用 CLI 的人）
└── npm install + 3 个核心命令演示

第五屏：数据结构（给想深入的开发者）
└── 一个节点 Markdown 示例，说明四层结构的意义
```

### 三行痛点的写法原则

- 用具体场景，不用抽象概念（"对齐失效"换成"AI 开始生成，但它没看到你脑子里的假设"）
- 每行都是一个可以被认领的经历——读者会说"对，这就是我"
- 不提工具名称，不提解决方案，只描述痛

## 讨论共识

- ✅ 一句话定位：**让 AI 知道你怎么想的，而不只是你要什么**
- ✅ 第一屏：通用痛点三行，不提 CLI，两类人都能感受
- ✅ 演示：先文字占位，后补真实对话截图序列
- ✅ 结构：痛点 → 演示 → 解决了什么 → 快速开始 → 数据结构
- ✅ 文字风格：具体场景 > 抽象概念，每句话都是可被认领的经历
