# 项目介绍——如何让不同受众立刻理解dt的价值

## 已知前提
- dt 的价值不只是“管理 Markdown 文件”，而是让 AI 和人共享推导过程。
- README、仓库描述和 Agent 入口文件面对不同受众，不能把所有协议细节塞进第一屏。
- 现有新版结构已经支持分布式节点：项目思想集中在 `项目思想/`，实现决策就近放在源码目录。

## 认知起点
介绍的首要目标不是完整解释功能，而是让读者迅速认领痛点：AI 做得很快，但它不知道人的假设、边界和判断依据；对话结束后，推导过程丢失，下一次协作又要重新对齐。

## 推导逻辑
- 先定义受众：开发者、AI 重度用户、GitHub 路人。
- 再定义痛点：上下文断裂、对齐失效、失去掌控感、思考被外包。
- README 用一句话抓住价值：让 AI 知道你怎么想的，而不只是你要什么。
- 技术细节往后放：本地 Markdown、CLI、前端图谱、分布式节点、`.dt/index.yaml`。
- 项目内 DT 节点采用新版分层结构，读者和 Agent 都能从文件位置看出节点所属语境。

## 讨论共识
- 介绍从痛点开始，不从命令或数据结构开始。
- 对外第一句话强调“思考过程”，而不是“决策树工具”这个抽象名词。
- README 已经补入截图、快速开始、分布式节点与数据模型说明。
- 这棵介绍子树继续保留在 `项目思想/`，作为对外叙事和项目定位的长期记忆。
