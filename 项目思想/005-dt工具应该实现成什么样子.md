# dt工具应该实现成什么样子

## 已知前提
- dt 的使用者有两类：AI Agent（通过 CLI）和人类（通过前端可视化 + 直接编辑文件）
- dt 本质是认知对齐 + 过程外化 + 参与控制工具
- 所有数据本地化（Markdown + YAML），不依赖外部服务

## 认知起点
确定了 dt 要做什么之后，需要讨论具体如何实现。实现形态直接影响工具的可用性和 Agent 的准确使用率。

## 推导逻辑
工具形态的核心约束是两类用户的工作方式不同，需要分别设计，然后统一在同一套数据模型上。

## 讨论共识
- ✅ CLI 是 Agent 接口，前端可视化是人类接口，共享同一套本地 DT 数据
- ✅ 数据本地化，节点是带 `dt: node/v1` frontmatter 的 Markdown 文件，可放在项目任意目录
- ✅ 多项目通过全局注册表 ~/.dt/projects.json 管理
- ✅ `.dt/` 现在承担项目配置与索引职责，`.dt/index.yaml` 记录 `node id -> 文件路径`
