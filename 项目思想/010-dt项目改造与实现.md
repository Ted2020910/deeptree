# dt项目改造与实现

## 已知前提（讨论共识汇总）

### 来自 [[009]] 数据模型
- 节点 = frontmatter（id/标题/摘要/类型/状态/edges）+ 自由 Markdown 正文
- 新增"摘要"字段
- 类型支持预设 + 自定义
- 边存储在节点 frontmatter 中（方向 + 摘要 + 可选详细逻辑链接）
- 支持跨分支关联，底层是有向图

### 来自 [[008]] CLI 命令设计
- 核心读写命令：tree / status / show / add / link / update
- 分布式节点维护：scan 自动发现带 `dt: node/v1` 的 Markdown
- 多项目管理：register / unregister / projects
- frontmatter 走 CLI/API，正文直接写回节点实际 Markdown 路径
- 前端是人类组织界面，支持文件树、新建节点、新建文件夹、拖拽连边和详情编辑

## 推导逻辑

### 当前已落地结构

| 层 | 当前实现 |
|---|---|
| **项目元数据** | `.dt/tree.yaml` 保存项目配置，`.dt/index.yaml` 保存 `node id -> path` |
| **节点文件** | 任意项目目录中的 Markdown，使用 `dt: node/v1` frontmatter 标记 |
| **关系模型** | `edges` 统一表达 `from/to`，系统维护双向边 |
| **CLI** | `tree/status/show/add/link/update/scan/register/projects` 覆盖查看、写入、索引和多项目管理 |
| **前端** | Project Explorer + React Flow Canvas + Detail Panel |
| **服务端** | `/api/projects/:id/*` 多项目 API，WebSocket 推送刷新 |
| **桌面端** | Electron 壳启动本地 HTTP/WS 服务并读取本地文件系统 |

## 讨论共识

- 新版核心数据模型已经落地，旧 `parent/children` 只作为读取时的兼容迁移输入。
- 节点类型保持自由字符串，模板只作为项目级 `dt` skill 的参考。
- 新节点可以先创建，再通过 CLI、详情面板或画布拖拽补关系。
- 分布式节点已经成为当前推荐结构：具体实现节点放在对应代码目录，项目思想节点集中在 `项目思想/`。
- `.dt/` 不再是节点正文仓库，而是项目配置与索引中心。
